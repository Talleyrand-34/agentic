# Agentic Library — Agent Instructions

This repository is a **centralized library of vendor-agnostic agentic configurations**.
It is never used directly by end-user agents. Its fragments are composed into target
project repositories via the tooling provided here.

## What This Repo Contains

- `agents/` — Composable AGENTS.md building blocks (fragments)
- `skills/` — Reusable agent skill definitions
- `automations/` — n8n workflow exports and templates
- `vendors/` — Vendor-specific adapter configurations
- `profiles/` — Composition presets for common project types
- `tooling/` — Scripts and utilities (called via `just`)
- `schemas/` — JSON Schemas for all structured files

## Rules for Working in This Repo

### File Authoring

- Fragment files in `agents/` must be **self-contained**: no cross-references to other fragments,
  no project-specific paths, no hardcoded credentials.
- Every fragment starts with a single H2 heading (`## Section Name`) that matches its purpose.
- Fragments must not exceed 300 lines. Split larger content into multiple focused fragments.
- Use `{TOKEN_NAME}` syntax for project-specific values that get substituted at compose time
  (e.g., `{BUILD_CMD}`, `{PROJECT_NAME}`).

### Skill Authoring

- Every skill directory must have a `SKILL.md` with valid YAML frontmatter.
- Frontmatter must include: `name`, `description`, `version`, `tags`, `vendor_support`.
- Skills must declare all supporting files in `resources:` frontmatter.
- Use SemVer for `version` (`MAJOR.MINOR.PATCH`).

### Vendor Adapters

- Never edit vendor-generated output files manually. They are derived artifacts.
- The canonical source of truth is always `AGENTS.md` (assembled from fragments).
- `adapter.json` files must validate against `vendors/_schema/adapter.schema.json`.

### Profiles

- Profile YAML files must validate against `schemas/profile.schema.json`.
- Profile names follow the pattern: `{language}-{framework-or-architecture}-{pattern}.yaml`.
- Profiles declare intent (which fragments), not implementation (how to assemble them).

### Tooling

- All user-facing commands are exposed as `just` recipes in the root `justfile`.
- Shell scripts in `tooling/lib/` are implementation details — not called directly.
- Scripts must be idempotent: running them twice with the same inputs produces identical output.

### Index Files

- `index/skills.json` and `index/fragments.json` are generated by `just index`.
- Always run `just index` after adding or modifying skills or fragments.
- The index files are committed to the repo so they are available without running tooling.

## Commit Conventions

Follow Conventional Commits format:
- `feat(fragments):` — new fragment added
- `feat(skills):` — new skill added
- `feat(profiles):` — new profile added
- `feat(vendors):` — new vendor adapter or template
- `fix(fragments):` — correction to an existing fragment
- `fix(tooling):` — bug fix in scripts
- `chore(index):` — index rebuild
- `chore(ci):` — CI/CD changes

## Commands Reference

```
just list-profiles          # List all available profiles
just list-skills            # List all available skills
just compose PROFILE TARGET # Assemble AGENTS.md into a target project directory
just vendor-gen TARGET      # Generate vendor-specific files from target's AGENTS.md
just deploy PROFILE TARGET  # compose + vendor-gen + deploy skills
just validate TARGET        # Validate an assembled AGENTS.md in a target project
just sync-check TARGET      # Check if target project's config has drifted from library
just index                  # Rebuild index/skills.json and index/fragments.json
just lint                   # Validate all fragments, profiles, and adapter files
```
